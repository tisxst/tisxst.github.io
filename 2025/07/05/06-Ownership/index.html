内存管理是一个编程语言的重要组成部分，很多其他语言采用的内存回收机制是扫描运行时的变量，没用的抛弃掉。但是 Rust 采用了独特的 ownership 机制，确保了内存安全，但是这对于程序员来说是一个全新的概念，需要重新了解。

在深入了解关于 ownership 的内容之前，先了解一下 Rust 和内存的简单知识。

1 - 内存堆栈

内存内部分为两个结构，也就是堆和栈。

栈英文为 stack，就好像是一个健身房中的杠铃，数据在栈中就是杠铃片，我们无法在取出前一片之前将后一片取出。因此栈中的数据遵循的原则是“后进先出”，也就是 Last in, first out 的原则。将数据存入栈中的操作叫做入栈（pushing onto the stack），从栈中移除数据叫出栈（popping off the stack）。所有存储在栈上的数据都必须是已知且大小固定的。那些在编译时大小未知或其大小可能发生变化的数据，则必须存储在堆上。

堆英文为 heap，数据存放没有那么多讲究，内存分配的时候发现哪里有空地就放哪里，然后将这个位置标识为已被占用，然后返回一个该地址的指针（pointer），这个玩意在 C 语言中是一个难点，我在研究 C 的时候第一个拦路虎就是指针。

由于指针（也就是分配到堆中的具体地址）是一个固定长度，固定类型的值，所以指针可以存在栈中（顺序问题后续再细说）。栈的存取速度比堆快得多，因为读取只需要按照顺序来就可以，第一个数据永远在栈的顶部；但是堆中的数据存储是要找空的内存地址，读取的时候要寻找指针的位置才可以。

调用函数时，各种数据会先压入栈中，然后读取完成再做出栈，这样程序运行效率就要比现从堆中找数据要快得多。

Rust 所采用的 ownership 方案则不需要我们考虑操作堆栈的繁琐，不过了解内存的运行原理是有利于我们理解这个“所有权”的概念的。

2 - 所有权的规则

  1. Rust 中每个值都有“主人”，也就是拥有者（owner）
  2. 每个值同时只能有一个主人
  3. 如果主人挂了，这个值是要陪葬的

2.1 - 字符串类型

通常来说，前边笔记中的基础类型的变量都是有固定大小的变量类型，所以绝大多数情况下这些变量时存在栈中的。字符串不一样，复杂一些。

对比一下如下两个字符串

这是两种不同的创建字符串的方式，他们之间有着很明显的区别。

let s = "Hello"; 创建了一个固定大小的字符串，且不可更改，该字符串会存在栈中调用速度很快，但是属于“硬编码”在二进制文件中了。如果硬改，编译器会报错。

let s = String::from("Hello"); 则是在堆中开辟了一个字符串实例，而栈中包含了变量 s。此时 s 并不储存 Hello 这个字符串，而是储存了一个指向堆中 Hello 字符串内存位置、长度和容量的结构体。

虽然堆中的这个字符串是可变的，但是 s 本身不可变，因此我们没法通过去修改 s 指向的字符串来改变堆中的值，编译器会报错。

这里需要额外解释一下 String::from 的作用。因为这里实际上是在堆中创建了一个实例，所以使用时双冒号 :: 来调用 from 方法。我们同时也会见到打点调用的方法，比如 s.push_str，此时的 push_str 是一个实例方法，只能在有实例的时候进行调用。

举个现实中的例子，如果我们有一个机器人，我门下的指令是 红烧茄子::做，此时机器人会去执行并生成一盘红烧茄子，而如果指令是 红烧茄子::吃，机器人是无法执行的，因为机器人没有办法在没有红烧茄子的时候吃掉不存在的红烧茄子，只能在执行完 ::做 之后，对已经做出来的红烧茄子执行 .吃。

对于 Rust 来说，打点调用的是实例方法，而使用路径分隔符（也就是双冒号）的方法为类方法。
